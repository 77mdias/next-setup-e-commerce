# S05-SEC-002 - Migrar reset de senha para token hash + consumo one-time

**Data:** 2026-03-11  
**Status:** ✅ Concluída

## Resumo

O fluxo de recuperação/redefinição de senha foi migrado para persistir somente hash de token (SHA-256) e consumir token com invalidação one-time atômica.

- `forgot-password` agora gera `{ token, tokenHash }`, persiste apenas `tokenHash` e envia apenas o token bruto por email.
- `reset-password` valida `tokenHash + expiração` e consome token via `updateMany` atômico, impedindo reuso concorrente.
- Resposta de erro foi unificada para token inválido, expirado ou já utilizado.

## Discovery técnico aplicado

- Fluxos ponta a ponta revisados:
  - `POST /api/auth/forgot-password` -> lookup de usuário + emissão de link de reset.
  - `POST /api/auth/reset-password` -> validação de token + atualização de senha.
  - UI `auth/reset-password` preservada no contrato (`email + token + newPassword`).
- Contratos entre camadas preservados:
  - Frontend continua consumindo os mesmos endpoints e status HTTP.
  - Payload de sucesso de reset permanece inalterado.
  - Mensagem genérica de `forgot-password` para anti-enumeração foi mantida.
- Riscos de segurança/regressão mapeados:
  - risco de vazamento de token por dump de banco (mitigado por hash).
  - risco de race condition no consumo de token (mitigado com consumo atômico por `updateMany`).
  - risco de quebra de token emitido antes do deploy (mitigado por migração SQL que converte valores legados para hash).

## Decisões técnicas

- Criado utilitário compartilhado `src/lib/secure-token.ts` para geração/hash de token seguro.
- `src/lib/email.ts` passou a reutilizar o utilitário para manter consistência de algoritmo entre verificação e reset.
- Campo Prisma renomeado semanticamente para `resetPasswordTokenHash` com `@map("resetPasswordToken")`:
  - mantém compatibilidade física da coluna existente;
  - reduz chance de regressão por escrita de plaintext na aplicação.
- Consumo one-time do reset feito com operação de escrita atômica:
  - evita dupla utilização em chamadas concorrentes com o mesmo token.

## Verificação manual de cenários críticos

- Cenário: emissão de reset para email existente.
  - Verificado em código/teste que apenas `resetPasswordTokenHash` é persistido e o link contém token bruto.
- Cenário: reset com token válido.
  - Verificado em teste de integração que a atualização de senha limpa token/expiração no mesmo write.
- Cenário: reuso de token.
  - Verificado em teste de integração via retorno `count = 0` com resposta unificada para inválido/expirado/já usado.
