# S02-ORD-001 - Ajustar /api/orders/[orderId] para ownership por userId

**Data:** 2026-03-01  
**Status:** ✅ Concluida

## Resumo

A rota `GET /api/orders/[orderId]` deixou de autorizar por `customerEmail` e passou a usar ownership canonico por `session.user.id` + `order.userId`, com validacao previsivel de `orderId` invalido.

## Implementacao aplicada

- API `src/app/api/orders/[orderId]/route.ts`:
  - Sessao agora exige `session.user.id` para autenticar (`401` quando ausente).
  - Adicionada normalizacao/validacao estrita de `orderId` (apenas inteiro positivo), retornando `400` quando invalido.
  - Consulta alterada para `findFirst({ where: { id, userId: session.user.id }})`.
  - Resposta para nao-owner permanece `404` generico sem exposicao de metadados.
- Testes `src/app/api/orders/[orderId]/__tests__/route.integration.test.ts`:
  - Ajuste dos cenarios de sucesso/404 para ownership por `userId`.
  - Novo teste para `orderId` invalido com `400` e sem consulta ao banco.
  - Novo teste explicito de nao-owner recebendo `404`.

## Impacto tecnico mapeado

- API: mudanca direta apenas em `GET /api/orders/[orderId]`.
- Frontend: sem mudanca de contrato de sucesso; consumidores passam a receber `400` para `orderId` invalido (ja tratado genericamente por erro de fetch).
- Schema: sem migracao; `Order.userId` ja existe (nullable), ficando alinhado para fase de backfill (`S02-ORD-002`).
- Integracoes: sem impacto em webhook/checkout; fluxos de sucesso/falha continuam via rota por `sessionId`.

## Validacoes executadas

- `npm run lint` ✅ (warnings pre-existentes de `react-hooks/exhaustive-deps`)
- `npm run build` ✅

## Riscos residuais

- Pedidos legados sem `userId` deixam de ser acessiveis por esta rota ate o tratamento da task `S02-ORD-002`.
