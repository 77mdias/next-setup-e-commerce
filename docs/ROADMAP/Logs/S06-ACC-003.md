# S06-ACC-003 - Garantir isolamento multi-store no escopo de `STORE_ADMIN`

**Data:** 2026-03-15  
**Status:** ✅ Concluída

## Resumo

O painel admin passou a resolver e aplicar escopo por loja para `STORE_ADMIN`, usando as lojas possuídas pelo usuário (`Store.ownerId`) como fonte de verdade no contexto atual do projeto. O catálogo admin agora filtra listagens por `storeId`, valida tentativas cross-store antes de acessar recursos e mantém `SUPER_ADMIN`/`ADMIN` com visão global.

- `requireAdminAccess()` agora injeta `adminStoreScope` no contexto autenticado.
- `GET /api/admin/products` foi criado com filtro obrigatório por escopo e suporte a validação explícita de `storeId`.
- `PUT /api/admin/products/[productId]/images` passou a bloquear persistência em produtos fora do escopo autorizado.
- O painel `/admin/remove-bg` deixou de consumir a API pública de catálogo e passou a usar o endpoint admin scoped.

## Discovery técnico aplicado

- Estado anterior:
  - O RBAC validava apenas papel/recurso/ação, sem escopo de loja.
  - `PUT /api/admin/products/[productId]/images` atualizava qualquer produto acessível por `id`, sem checar `storeId`.
  - `/admin/remove-bg` listava produtos via `/api/products`, herdando o store ativo público em vez do escopo do operador.
- Contratos relevantes:
  - O schema já modelava `Store.ownerId`, permitindo derivar o conjunto de lojas operadas pelo usuário sem nova tabela de vínculo.
  - `isAdminRole()` e a camada RBAC já previam `STORE_ADMIN`/`SUPER_ADMIN`, mas o enum Prisma ainda não materializava esses papéis.
- Riscos identificados:
  - Vazamento de catálogo entre lojas na UI e na API admin.
  - Mutação cross-store por `productId` conhecido.
  - Divergência entre papel administrativo e escopo efetivamente autorizado.

## Decisões técnicas

- Materializado o enum `UserRole` com `SUPER_ADMIN` e `STORE_ADMIN`, acompanhado de migration versionada.
- Adicionado `src/lib/admin-store-scope.ts` para:
  - normalizar papéis admin compatíveis com escopo;
  - resolver o escopo de `STORE_ADMIN` a partir de `Store.ownerId`;
  - reutilizar as checagens de acesso por `storeId`.
- Mantido o modelo de escopo simples e suficiente para demo/portfolio:
  - `STORE_ADMIN` enxerga somente as lojas de que é owner;
  - `ADMIN` e `SUPER_ADMIN` seguem com escopo global para evitar regressão funcional.
- Reforçada observabilidade de segurança com evento `admin.store_scope.denied`, usando logger estruturado e redaction já existente.

## Riscos remanescentes e mitigação

- Risco: pedidos e clientes admin ainda não existem no código atual, então o escopo compartilhado ainda não foi exercitado nesses recursos.
  - Mitigação: o contrato de escopo já está centralizado em `auth/rbac` para ser consumido pelas próximas rotas de `orders` e `customers`.
- Risco: o vínculo por `Store.ownerId` é suficiente para o estado atual, mas não suporta cenários futuros de operadores de loja não proprietários.
  - Mitigação: a abstração `admin-store-scope` isola essa decisão e pode migrar para uma tabela de memberships sem refatorar as rotas admin.
