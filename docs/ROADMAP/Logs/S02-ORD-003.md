# S02-ORD-003 - Ajustar consumidores frontend para novo contrato de ownership

**Data:** 2026-03-01  
**Status:** ✅ Concluida

## Resumo

Consumidores frontend de pedidos foram alinhados ao contrato de ownership canonico por `userId`, com tratamento seguro de `400/401/404` e recuperacao de UX sem exposicao de dados de pedidos inacessiveis.

## Implementacao aplicada

- `useCheckout` passou a mapear erros de `/api/orders/:orderId` com mensagens neutras e acao de recuperacao:
  - `400` -> pedido invalido (`open-orders`);
  - `401` -> sessao expirada (`reauth`);
  - `404` -> pedido nao disponivel para a conta (`open-orders`);
  - demais status -> fallback com `retry`.
- `OrdersPageContent` agora valida o `orderId` vindo do redirecionamento (`/orders?orderId=...`) usando `/api/orders/:orderId` antes do carregamento filtrado:
  - em `401`, redireciona para fluxo de autenticacao;
  - em `400/404`, mostra feedback neutro e acao de recuperacao (`View all orders`) sem exibir dados de pedido.
- Fluxo de recuperacao foi adicionado ao banner de erro da pagina de pedidos para limpar filtro e voltar para `/orders` quando o pedido alvo nao e acessivel.
- Testes de integracao de `orders/success` e `orders/failure` foram ampliados para cobrir normalizacao de `session_id`, preservando redirecionamento seguro no fluxo pos-checkout.
