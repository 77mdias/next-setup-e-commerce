# S06-DSH-002 - Construir shell de navegação administrativa e página inicial do dashboard

## Resumo

Foi entregue a primeira camada visual do painel administrativo em `/admin`, com shell dedicado, navegação estável entre módulos, resumo do contexto operacional e página inicial do dashboard preparada para a próxima integração de KPIs.

O painel agora deixa de herdar o chrome público e passa a operar em uma superfície própria para uso interno, protegida pelo `layout` server-side já alinhado ao middleware de admin.

## Estado anterior

- O acesso a `/admin` já era protegido, mas a rota não possuía shell visual nem landing page operacional.
- O `RootLayout` continuava injetando navegação e footer públicos em qualquer rota, inclusive no contexto administrativo.
- Não existiam páginas canônicas para `orders`, `catalog`, `customers` e `audit` dentro de `/admin`, o que deixaria links quebrados ao introduzir a sidebar.

## Decisões técnicas

- Criado `AdminShell` com:
  - breadcrumbs;
  - resumo do operador, papel e escopo;
  - estrutura responsiva para desktop e mobile.
- Criado `AdminSidebar` com:
  - links estáveis para `dashboard`, `orders`, `catalog`, `customers` e `audit`;
  - estado ativo por rota;
  - resumo persistente do contexto administrativo.
- O `layout` de `/admin` agora resolve contexto de shell a partir do operador autenticado:
  - `ADMIN` e `SUPER_ADMIN` recebem rótulo de visão global;
  - `STORE_ADMIN` consulta as lojas do escopo para exibir contexto real no shell.
- O chrome público passou a ser omitido em rotas `/admin`.
  - Motivo: separar claramente experiência pública e operação interna, evitando dupla navegação e ruído visual no painel.
- Foram criadas páginas placeholder para módulos ainda não implementados.
  - Motivo: cumprir a aceitação de navegação sem links quebrados antes das tasks operacionais de pedidos, catálogo, clientes e auditoria.
