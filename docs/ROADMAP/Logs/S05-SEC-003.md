# S05-SEC-003 - Padronizar invalidacao e limpeza de tokens expirados

**Data:** 2026-03-11  
**Status:** ✅ Concluída

## Resumo

Foi padronizado o ciclo de vida de tokens de autenticacao com regras determinísticas para expiracao, invalidacao e limpeza.

- TTL de tokens foi centralizado:
  - verificacao de email: 24h;
  - reset de senha: 1h.
- Consumo de token de verificacao passou a usar invalidacao atomica (`updateMany`) para evitar reuso concorrente.
- Tokens expirados agora sao limpos explicitamente no fluxo de leitura (tentativa de uso de token expirado) e no fluxo de emissao/reenvio.
- Foi adicionada rotina opcional para limpeza periodica em lote: `npm run tokens:cleanup-expired`.

## Discovery tecnico aplicado

- Impacto em API:
  - `GET /api/auth/verify-email`: consumo one-time reforcado com escrita atomica e limpeza de expirados por hash quando token nao e mais valido.
  - `POST /api/auth/verify-email`: limpeza de expirados por usuario antes do reenvio e sobrescrita do hash/expiracao para invalidar token anterior.
  - `POST /api/auth/forgot-password`: limpeza de reset expirado por usuario antes de emitir novo token.
  - `POST /api/auth/reset-password`: limpeza do token expirado correspondente quando tentativa de consumo falha por expiracao/reuso.
  - `POST /api/auth/register`: passou a consumir TTL centralizado para evitar divergencia de expiracao entre pontos de emissao.
- Impacto em frontend:
  - nenhum ajuste de contrato foi necessario; payloads/status HTTP permaneceram compativeis.
- Impacto em schema:
  - sem mudanca de schema/migracao nesta task.
- Impacto em integracoes:
  - sem mudanca de provider externo; fluxo SMTP preservado.

## Decisoes tecnicas

- Criado modulo compartilhado `src/lib/auth-token-lifecycle.ts` para:
  - constantes oficiais de TTL;
  - funcoes de expiracao;
  - funcoes de limpeza de token expirado (escopo por usuario/email/hash);
  - limpeza global opcional em lote.
- Regras de invalidacao documentadas na Sprint 05 para reduzir ambiguidades operacionais.
- Criado script operacional `scripts/cleanup-expired-auth-tokens.js` com suporte a `dry-run` e modo `--apply`.

## Riscos remanescentes e mitigacao

- Risco: limpeza periodica em lote depende de agendamento externo.
  - Mitigacao: script operacional pronto (`npm run tokens:cleanup-expired`); pendente apenas plugar em cron/job do ambiente.
- Risco: transporte SMTP ainda usa `rejectUnauthorized: false`.
  - Mitigacao: hardening previsto em `S05-HRD-001`.
