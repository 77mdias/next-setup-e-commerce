# S04-RLS-001 - Tornar pipeline CI bloqueante com lint, build e testes críticos

**Data:** 2026-03-11  
**Status:** ✅ Concluída

## Resumo

O pipeline de CI foi consolidado como gate crítico de release com validação rastreável de:

- `npm run lint`
- `npm run build`
- `npm run test:unit:critical`
- `npm run test:integration:critical`
- `npm run test:e2e:critical:ci`

Também foi removida redundância de build no fluxo E2E de CI, reduzindo tempo total de execução sem alterar o comportamento do gate.

## Discovery técnico aplicado

### Estado encontrado

- Workflow já executava checks de qualidade e testes críticos no job `Quality Gates`.
- O comando de E2E (`test:e2e`) reconstruía a aplicação dentro da própria etapa, gerando build duplicado porque o CI já havia executado `npm run build` antes.

### Mapeamento de impacto real

- API: sem alteração de contrato.
- Frontend: sem alteração de comportamento em runtime.
- Schema/Prisma: sem alteração.
- Integrações externas (Stripe/Auth/Webhooks): sem alteração funcional.
- CI/Release: ajuste de orquestração para gate mais eficiente e rastreável.

## Implementação

### 1) Package scripts

- `package.json`:
  - adicionado `test:e2e:critical:ci` (prepara dados E2E + executa Playwright crítico **sem** rebuild).
  - mantido `test:e2e` para uso local completo (com build).

### 2) Pipeline CI

- `.github/workflows/ci.yml`:
  - `timeout-minutes` do job `Quality Gates` ajustado de `30` para `40` para reduzir risco de timeout em execução crítica completa.
  - etapa `E2E critical suite` passou a usar `npm run test:e2e:critical:ci`.

### 3) Governança de release

- Em branch protegida, o status check do workflow `CI / Quality Gates` deve permanecer como obrigatório para manter bloqueio automático de merge em falha de qualquer etapa crítica.

## Critérios de aceite

| Critério de aceitação                                                 | Evidência                                                                                 | Status |
| --------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- | ------ |
| Merge bloqueado automaticamente quando qualquer teste crítico falhar  | Job único `Quality Gates` falha em qualquer etapa crítica e expõe status check bloqueante | ✅     |
| Pipeline executa lint, build e suíte crítica com resultado rastreável | Etapas nomeadas no workflow + logs por comando crítico                                    | ✅     |

## Validações executadas

- ✅ `npm run lint`
- ✅ `npm run build`
- ✅ `npm run test:unit:critical`
- ✅ `npm run test:integration:critical`

Observação operacional:

- ⚠️ `npm run test:e2e:critical:ci` foi executado localmente e falhou por indisponibilidade do banco externo (`PrismaClientInitializationError` ao conectar no host Neon). O comando/scripting está válido, porém a validação E2E ponta a ponta depende de ambiente CI com serviço de banco provisionado.

## Riscos residuais e mitigação

- Risco: instabilidade transitória de ambiente CI (ex.: banco/runner/browser) aumentar duração da etapa E2E.
  - Mitigação: timeout do job ampliado para `40` minutos e manutenção de retries no preparo E2E.
- Risco: check obrigatório não configurado em branch protegida em algum repositório espelho.
  - Mitigação: manter `CI / Quality Gates` como required status check nos branches de release.
