# S06-ACC-002 - Implementar matriz RBAC por recurso/ação nas APIs administrativas

**Data:** 2026-03-14  
**Status:** ✅ Concluída

## Resumo

As APIs administrativas passaram a usar uma camada RBAC compartilhada por recurso/ação, com negação por padrão, contrato uniforme de erro (`401/403`) e logging estruturado/redigido para eventos de autorização.

- `src/lib/rbac.ts` centraliza a política para `dashboard`, `orders`, `catalog` e `customers`.
- As rotas admin existentes (`/api/admin/remove-bg` e `/api/admin/products/[productId]/images`) agora usam o mesmo fluxo de autorização e o mesmo payload de erro.
- O helper compartilhado de `remove-bg` legado/admin também passou a propagar o contexto RBAC nos logs para manter observabilidade consistente.

## Discovery técnico aplicado

- Estado anterior:
  - `requireAdminAccess()` fazia apenas um check binário de "admin vs não-admin".
  - Cada rota admin montava seu próprio `401/403`, com mensagens diferentes e sem política por recurso/ação.
  - A rota de persistência de imagens usava `console.error`, fugindo do logger estruturado do projeto.
- Contratos relevantes:
  - `isAdminRole()` já aceitava `ADMIN`, `SUPER_ADMIN` e `STORE_ADMIN`, mesmo com o schema ainda limitado a `ADMIN`.
  - `createRequestLogger()` e `log-redaction.ts` já forneciam a base para observabilidade segura.
- Riscos identificados:
  - Divergência de regra entre rotas admin conforme novos endpoints fossem surgindo.
  - Respostas de autorização inconsistentes, facilitando regressão e dificultando consumo uniforme pelo frontend.
  - Perda de correlação/telemetria em falhas sensíveis por uso de `console.error`.

## Decisões técnicas

- Criado `src/lib/rbac.ts` com:
  - política por `resource/action`;
  - mapeamento opcional `HTTP method -> action`;
  - contrato uniforme de erro com `code` estável (`ADMIN_AUTH_REQUIRED`, `ADMIN_ACCESS_DENIED`);
  - logging de negação com `securityDomain=admin_rbac`, `adminResource`, `adminAction`, `adminRole` e `userId`.
- Mantido `requireAdminAccess()` como guarda binária base para sessão/papel administrativo, e movida a granularidade para a nova camada RBAC.
- Migrado `src/lib/remove-bg-route-security.ts` para usar a autorização RBAC compartilhada e preservar o contexto autorizado nos logs.
- Migrada `src/app/api/admin/products/[productId]/images/route.ts` para usar `createRequestLogger()` + RBAC, removendo `console.error`.
- Definida política conservadora para evolução futura:
  - `dashboard.read`, `customers.read`, `orders.read/update`, `catalog.read/update` liberados para `STORE_ADMIN`, `ADMIN`, `SUPER_ADMIN`;
  - ações destrutivas (`delete`) ficam mais restritas por padrão.

## Riscos remanescentes e mitigação

- Risco: o schema Prisma atual ainda não materializa `SUPER_ADMIN` e `STORE_ADMIN`.
  - Mitigação: a política já está pronta para esses papéis, mas a ativação efetiva dependerá da evolução de identidade/escopo nas próximas tasks da Sprint 06.
- Risco: a cobertura E2E crítica do repositório depende de banco remoto externo e falhou por indisponibilidade de infraestrutura.
  - Mitigação: build, typecheck, lint, suites críticas e integrações focadas do escopo passaram; o bloqueio foi isolado e documentado com erro concreto.
- Risco: apenas duas rotas admin existem hoje, então a matriz ainda não foi exercitada em `orders`, `customers` e `dashboard`.
  - Mitigação: o módulo compartilhado já define a política desses recursos e as próximas rotas da Sprint 06 devem consumir o helper em vez de reimplementar autorização.
