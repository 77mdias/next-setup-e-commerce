# S04-TST-003 - Introduzir cenário E2E de compra para validação de release

**Data:** 2026-03-10  
**Status:** ✅ Concluída

## Resumo

Foi implementado o cenário crítico E2E de compra com Playwright cobrindo:

- add-to-cart em UI real
- checkout com retorno de sucesso (`/orders/success -> /orders?orderId=...`)
- fallback de falha de pagamento (`/orders/failure -> /orders?orderId=...&checkout=failed`)

Além da suíte, o CI foi atualizado para executar E2E de forma bloqueante em PR/release e publicar artefatos (`html report`, `trace`, `video`, `screenshot`) para diagnóstico.

## Discovery técnico aplicado

- Fluxo ponta a ponta analisado no código:
  - `src/context/cart.tsx`
  - `src/app/carrinho/page.tsx`
  - `src/app/checkout/page.tsx`
  - `src/hooks/useCheckout.ts`
  - `src/app/api/checkout/route.ts`
  - `src/app/orders/success/page.tsx`
  - `src/app/orders/failure/page.tsx`
  - `src/components/orders/orders-page-content.tsx`
- Contrato entre camadas:
  - UI checkout consome `POST /api/checkout` e usa `url` retornada para redirecionar.
  - `orders/success|failure` dependem de `session_id` mapeado por `stripeCheckoutSessionId|stripePaymentId`.
  - Lista de pedidos em `/orders` depende de ownership por sessão autenticada.
- Risco principal identificado para CI:
  - Dependência externa de Stripe real tornava E2E não determinístico.

## Implementação

- Modo controlado para E2E no checkout backend:
  - `src/app/api/checkout/route.ts`
  - Novo modo `E2E_CHECKOUT_MOCK_MODE=true` com saída determinística sem chamada externa do Stripe.
  - Header técnico `x-e2e-checkout-outcome: failed` permite simular fallback de pagamento.
  - Persistência de estado e `statusHistory` para manter contrato real de leitura em `/orders`.
- Cobertura de integração para o novo modo:
  - `src/app/api/checkout/__tests__/route.integration.test.ts`
  - Cenários cobrindo sucesso/falha em modo E2E e ausência de chamada Stripe.
- Suíte Playwright crítica:
  - `e2e/checkout-critical-flow.spec.ts`
  - Dois cenários `@critical`:
    - compra crítica com retorno e status visível
    - fallback de falha com contexto de cancelamento visível
- Ambiente determinístico de dados E2E:
  - `scripts/prepare-e2e-data.js`
  - Provisiona usuário ativo, loja, produto e inventário; limpa carrinho/pedidos do usuário E2E.
- Pipeline CI bloqueante e artefatos:
  - `.github/workflows/ci.yml`
  - Serviço PostgreSQL, `prisma migrate deploy`, `npm run e2e:prepare`, instalação de Chromium e upload de artefatos Playwright.
- Ajustes de execução/config:
  - `playwright.config.ts`
  - `package.json`
  - `.env.example`

## Riscos residuais e mitigação

- Risco: modo de checkout simulado ser habilitado fora de ambiente de teste.
  - Mitigação: `E2E_CHECKOUT_MOCK_MODE` desligado por padrão (`false` no `.env.example`) e habilitado explicitamente apenas no fluxo E2E.
- Risco: variação de dados em ambientes compartilhados.
  - Mitigação: script `e2e:prepare` idempotente e limpeza de estado do usuário E2E antes da execução.

## Hardening pós-entrega (2026-03-10)

- Erro CI `tsc --noEmit` com `page`/`route` em `unknown`:
  - Causa-raiz: uso de shim local (`types/playwright-test.d.ts`) com `any`, mascarando ausência do pacote oficial e degradando inferência de fixtures do Playwright.
  - Correção: remoção do shim e tipagem explícita em `e2e/checkout-critical-flow.spec.ts` com `Page`, `Route` e payload tipado de `/api/products`.
- Erro CI `Cannot find module '@playwright/test'` / `playwright/test`:
  - Causa-raiz: runtime Playwright instalado de forma ad-hoc no workflow, sem dependência versionada do projeto.
  - Correção: inclusão de `@playwright/test` em `devDependencies` e remoção da etapa `npm install --no-save @playwright/test` do CI.
- Falha por seletor ambíguo de senha:
  - Causa-raiz: `getByLabel("Password")` colidindo com input e botão de visibilidade.
  - Correção consolidada: seletor estrito por role (`getByRole("textbox", { name: /^Password$/ })`).
- Timeout no clique de `View Product`:
  - Causa-raiz: dependência de renderização/listagem UI para navegar ao produto em ambiente CI.
  - Correção consolidada: resolução determinística do `productId` via `GET /api/products` e navegação direta para `/product/:id`.
