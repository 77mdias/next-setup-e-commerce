# S03-QA-001 - Cobrir integracao da API de enderecos (auth, CRUD e default)

**Data:** 2026-03-07  
**Status:** ✅ Concluida

## Resumo

A cobertura de integracao da API `/api/addresses` foi reforcada para reduzir risco de regressao em autenticacao, ownership, CRUD e regras de endereco padrao (`isDefault`). O contrato da rota foi mantido, com foco em endurecer validacao automatizada dos cenarios criticos.

## Implementacao aplicada

- Expandida a suite `src/app/api/addresses/__tests__/route.integration.test.ts` (de 9 para 15 cenarios).
- Cobertos acessos nao autenticados para todos os metodos (`GET/POST/PUT/DELETE`) com retorno `401` e sem mutacoes no banco.
- Coberto POST com usuario que ja possui enderecos para garantir que novo endereco pode permanecer `isDefault=false` sem derrubar default existente.
- Coberto PUT para caso limite em que endereco default recebe `isDefault=false` sem endereco substituto, preservando consistencia (mantem um default valido).
- Coberto DELETE de endereco nao-default assegurando ausencia de mutacoes desnecessarias na flag `isDefault`.
- Task atualizada em `docs/development/tasks/PHASE-03-experiencia-conta-pedido.md` com item e criterios marcados como concluidos.

## Validacoes

## Riscos residuais

- A suite usa mocks de `db` e `getCurrentUser`; embora valide contrato e regras de negocio da rota, ainda nao substitui validacao E2E com banco real e sessao real em ambiente de homologacao.
