# S05-SEC-001 - Migrar token de verificação de email para geração criptográfica com persistência por hash

**Data:** 2026-03-11  
**Status:** ✅ Concluída

## Resumo

O fluxo de verificação de email foi migrado para token criptográfico com persistência apenas do hash SHA-256 no banco.

- `register` agora gera `{ token, tokenHash }`, persiste apenas `tokenHash` e envia o token bruto por email.
- `verify-email (POST)` (reenvio) segue o mesmo padrão.
- `verify-email (GET)` valida por hash, mantém expiração e invalida o token no consumo (one-time use).

## Discovery técnico aplicado

- Fluxos ponta a ponta revisados:
  - `POST /api/auth/register` -> criação de usuário inativo + envio de link de verificação.
  - `GET /api/auth/verify-email` -> ativação de conta por token.
  - `POST /api/auth/verify-email` -> reenvio de token para conta inativa.
- Contratos entre camadas preservados:
  - Frontend continua consumindo os mesmos endpoints e status HTTP.
  - Payload de sucesso do `GET` foi mantido e passou a incluir `email` para robustez de redirecionamento.
  - Login por credenciais continua bloqueando usuários não ativos via `isActive` em `authOptions`.
- Riscos de segurança/regressão mapeados:
  - risco de vazamento de token em caso de dump de banco (mitigado por hash).
  - risco de quebra de links pendentes pré-migração (mitigado por migração SQL que converte plaintext para hash).

## Decisões técnicas

- Utilitário central de token em `src/lib/email.ts`:
  - `generateVerificationToken()` com `crypto.randomBytes(32)`.
  - `hashVerificationToken()` com SHA-256.
  - `generateVerificationTokenPair()` para uso consistente entre rotas.
- Campo Prisma renomeado semanticamente para `emailVerificationTokenHash` com `@map("emailVerificationToken")`:
  - mantém compatibilidade física da coluna no banco.
  - reduz chance de regressão por persistência acidental de plaintext na camada de aplicação.
- Migração SQL com `pgcrypto` para converter tokens legados em hash e adicionar índice de lookup.
