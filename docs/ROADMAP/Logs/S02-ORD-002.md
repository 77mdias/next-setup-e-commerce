# S02-ORD-002 - Definir estrategia para pedidos legados sem userId

**Data:** 2026-03-01  
**Status:** ✅ Concluida

## Resumo

Foi definida e implementada uma estrategia de transicao em duas camadas para pedidos legados sem `userId`:

- Backfill controlado por script (`customerEmail` -> `userId`) para vincular registros elegiveis.
- Fallback temporario na API `GET /api/orders/[orderId]` para auto-vincular o pedido legado no primeiro acesso autenticado do dono.

A resposta ao cliente permanece neutra (`404`) quando nao ha ownership valido, sem vazamento de dados.

## Implementacao aplicada

- API `src/app/api/orders/[orderId]/route.ts`:
  - Mantem ownership canonico por `{ id, userId: session.user.id }`.
  - Adiciona fallback temporario (com prazo de remocao ate **2026-03-31**) para pedido legado com `userId = null` e `customerEmail` igual ao email da sessao (case-insensitive).
  - Ao casar email, executa `updateMany` protegido para vincular `userId` e refaz lookup canonico.
  - Quando nao e possivel vincular, retorna `404` generico e registra log estruturado para revisao manual.

- Testes `src/app/api/orders/[orderId]/__tests__/route.integration.test.ts`:
  - Cobertura do fluxo de auto-vinculo legado elegivel.
  - Cobertura do fluxo sem correspondencia com `404` + rastreio para acao manual.

- Script operacional `scripts/backfill-order-user-id.js` (novo):
  - `dry-run` por padrao.
  - `--apply` para persistir alteracoes.
  - `--limit` para execucao controlada em lotes.
  - `--report <arquivo.md>` para gerar relatorio auditavel com casos pendentes.

- `package.json`:
  - Novo comando `npm run orders:backfill-user-id`.

## Politica de fallback e remocao

- Fallback por email e **temporario** e deve ser removido ate **2026-03-31**, apos:
  - execucao do backfill em producao,
  - saneamento dos casos pendentes,
  - confirmacao de ausencia de pedidos com `userId = null` acessiveis por usuario final.

## Validacoes tecnicas executadas

- `npm run lint` ✅
- `npm run build` ✅

## Riscos residuais

- Enquanto o fallback existir, ainda ha dependencia transitiva de `customerEmail` para pedidos legados.
- Casos sem email de pedido, sem email de sessao ou sem correspondencia de usuario exigem tratamento operacional manual via relatorio/log.
