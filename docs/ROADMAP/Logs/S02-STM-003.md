# S02-STM-003 - Expor historico e estado consistente nas APIs de pedido

**Data:** 2026-03-01  
**Status:** ✅ Concluida

## Resumo

As APIs de pedidos agora retornam `statusHistory` normalizado e consistente com o estado atual do pedido, incluindo fallback para pedidos legados sem trilha completa de historico.

## Implementacao aplicada

- Utilitario central (`src/lib/order-status-history.ts`):
  - Ordenacao temporal deterministica por `createdAt` e `id`.
  - Fallback para pedido legado sem historico (`reason:legacy_missing_history`).
  - Fallback para divergencia entre ultimo historico persistido e `order.status` atual (`reason:state_snapshot_mismatch`).
  - Saida padronizada para API com `createdAt` serializado e flag `isFallback`.
- API de pedido por ID (`src/app/api/orders/[orderId]/route.ts`):
  - Inclusao de `statusHistory` no `findFirst`.
  - Exposicao de `statusHistory` normalizado na resposta formatada, mantendo campos existentes.
- API de pedido por sessao (`src/app/api/orders/session/[sessionId]/route.ts`):
  - Inclusao de `statusHistory` no `findFirst`.
  - Resposta passa a retornar historico normalizado sem alterar o contrato base da rota.
- API de pedidos do usuario (`src/app/api/orders/user/route.ts`):
  - Inclusao de `statusHistory` no `findMany` paginado.
  - Normalizacao por pedido antes do retorno, preservando estrutura de paginacao e campos ja consumidos.
