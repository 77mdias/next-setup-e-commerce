# S05-HRD-003 - Hardening de `/api/remove-bg` com autenticação, limite de taxa e logs seguros

## Resumo

O hardening de `remove-bg` foi concluído para eliminar uso anônimo de uma operação custosa, alinhar o endpoint legado ao fluxo administrativo real e padronizar observabilidade segura. As rotas `POST/PUT /api/remove-bg` e `POST/PUT /api/admin/remove-bg` agora exigem sessão autenticada com role `ADMIN`, compartilham rate limit dedicado por IP e usuário autenticado, e registram falhas via logger estruturado sem expor payload sensível.

## Mudanças implementadas

- Criado helper compartilhado `src/lib/remove-bg-route-security.ts`:
  - valida acesso admin com `requireAdminAccess()`;
  - aplica rate limit compartilhado entre rotas `remove-bg` para evitar bypass alternando entre endpoints;
  - centraliza logging seguro de falhas operacionais com `createRequestLogger(...)`.
- Atualizada rota legada `src/app/api/remove-bg/route.ts`:
  - passou a exigir autenticação/autorização explícita antes de qualquer chamada ao provedor externo;
  - retorna `401` sem sessão, `403` sem role `ADMIN` e `429` com `Retry-After` quando excede o limite;
  - removeu `console.*` e passou a emitir logs estruturados (`warn/error`) com contexto mínimo e sem URL/token bruto.
- Atualizada rota administrativa `src/app/api/admin/remove-bg/route.ts`:
  - passou a usar o mesmo guard compartilhado de auth e rate limit;
  - migrou logs operacionais de `console.*` para logger estruturado;
  - adicionou warning explícito para falha parcial em processamento em lote.
- Alinhados defaults do frontend:
  - `src/hooks/useRemoveBg.ts` e `src/components/RemoveBgProcessor.tsx` agora defaultam para `/api/admin/remove-bg`, reduzindo risco de uso acidental do endpoint legado.
- Documentação atualizada:
  - `docs/04-setup-e-integracoes/REMOVE_BG_INTEGRATION.md`;
  - `docs/01-analise-aplicacao/03-rotas-ui-api.md`;
  - `docs/development/tasks/PHASE-05-hardening-identidade-seguranca.md`.

## Riscos remanescentes e mitigação

- O rate limit permanece em memória de processo, adequado ao escopo atual de demo/portfolio, mas não é distribuído em múltiplas réplicas.
  - Mitigação recomendada: migrar storage do rate limit para Redis/Upstash se houver escala horizontal.
- O script global `npm run format:check` continua falhando por configuração preexistente do repositório com `prisma/migrations/*.sql`, fora do escopo desta task.
  - Mitigação aplicada: validação de Prettier executada explicitamente apenas nos arquivos alterados.
