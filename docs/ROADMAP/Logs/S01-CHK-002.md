# S01-CHK-002 - Recalcular precos no servidor com dados canonicos

**Data:** 2026-02-28  
**Status:** ✅ Concluida

## Resumo

O fluxo de checkout server-authoritative foi validado e reforcado com cobertura de integracao para cenarios de produto inativo e produto fora da loja selecionada.
Os totais (`unitPrice`, `subtotal`, `shippingFee`, `total`) seguem derivados apenas de dados canonicos carregados do banco e os `line_items` Stripe usam valores em centavos calculados no backend.

## Implementacao aplicada

- Validacao de contrato de checkout sem preco do cliente mantida com schema estrito.
- Recálculo canônico no backend mantido para:
  - precos por produto/variacao (`unitPrice`, `unitPriceCents`);
  - subtotal, frete e total (`subtotalCents`, `shippingFeeCents`, `totalCents`);
  - payload de `line_items` Stripe com `unit_amount` canônico.
- Cobertura de testes adicionada em `src/app/api/checkout/__tests__/route.integration.test.ts`:
  - `returns 400 for inactive product and does not create order`;
  - `returns 400 when product does not belong to selected store`.

## Validacoes executadas

- `npm run test:integration` ✅ (32 testes aprovados)
- `npm run lint` ✅
- `npm run build` ✅

## Criterios de aceite atendidos

- [x] Totais do pedido sempre derivados de dados do banco.
- [x] Produto inválido/inativo/fora da loja retorna 400/404 e não cria pedido.

## Riscos residuais

- O endpoint ainda possui logs operacionais extensos em cenarios de erro; em producao, manter revisao de sanitizacao e nivel de log para evitar ruido operacional.
