# S06-ACC-001 - Criar guard de autenticação/autorização para `/admin`

**Data:** 2026-03-13  
**Status:** ✅ Concluída

## Resumo

A proteção de páginas administrativas foi reforçada para funcionar em duas camadas: `middleware` (edge) e `layout` server-side da árvore `/admin`.

- Usuário anônimo é redirecionado para fluxo de autenticação com callback seguro para `/admin`.
- Usuário autenticado sem papel administrativo recebe bloqueio consistente (`forbidden`) sem exposição de regra interna.
- A regra de papel admin foi centralizada em helper compartilhado para evitar divergência entre camadas.

## Discovery técnico aplicado

- Fluxo atual antes da task:
  - `/admin/**` dependia apenas de validação no `src/middleware.ts`.
  - Não existia `src/app/admin/layout.tsx`, então não havia guarda server-side na própria árvore de páginas.
- Contratos existentes:
  - Feedback de acesso usa `buildAccessFeedbackPath()` e página `/status`.
  - `requireAdminAccess()` era usado em endpoints admin, validando apenas `UserRole.ADMIN`.
- Riscos identificados:
  - Dependência exclusiva do middleware para páginas admin.
  - Divergência futura de regra de papel entre middleware e backend com evolução para `SUPER_ADMIN`/`STORE_ADMIN`.
  - Necessidade de garantir callback sem open redirect em fluxo de login admin.

## Decisões técnicas

- Criado `src/lib/admin-role.ts` para centralizar `isAdminRole()` e reutilizar em middleware + auth.
- Evoluído `src/lib/callback-url.ts` com `normalizeAdminCallbackPath()` para restringir callback admin ao escopo `/admin`.
- Evoluído `src/lib/auth.ts` com `resolveAdminPageAccess()` para contrato único de guarda de página (`auth-required`/`forbidden`).
- Criado `src/app/admin/layout.tsx` aplicando guarda server-side para toda a árvore `/admin/**`.
- Ajustado `src/middleware.ts` para usar os mesmos helpers de papel/callback admin e manter consistência com layout.
- Incluído `AIDEV-CRITICAL` no trecho central de decisão do guard em `auth.ts`.

## Riscos remanescentes e mitigação

- Risco: nesta etapa, a autorização admin continua binária (admin vs não-admin), sem matriz RBAC por recurso/ação.
  - Mitigação: guard de página centralizado e consistente já evita acesso indevido ao `/admin`; granularidade de permissões fica fora do escopo desta task e será tratada em task dedicada.
- Risco: não há cenário E2E exclusivo do painel admin nesta entrega.
  - Mitigação: foram executados testes de integração focados no guard (`layout` + middleware/rotas admin existentes), verificação manual de redirects críticos e E2E crítico de regressão global para garantir ausência de quebra transversal.
